# 04 — Interrupts: letting hardware (and errors) call our code

**What we built:** the **IDT** (Interrupt Descriptor Table) and the machinery behind
it — the system that lets the CPU **stop whatever it's doing and jump into our code**
when something happens: a programming error, a hardware event, or a deliberate signal.
This is arguably the most important plumbing in the whole OS. Almost everything later
(keyboard, mouse, the timer that drives multitasking) rides on top of it.

## The big idea: interrupts

Normally the CPU runs your instructions one after another. An **interrupt** is a way
to say "drop everything *right now* and go handle this." There are three sources:

1. **Exceptions** — the CPU itself raises these when something goes wrong:
   divide-by-zero, an illegal instruction, accessing forbidden memory, etc. Vectors
   **0–31** are reserved for these.
2. **Hardware interrupts (IRQs)** — devices raise these to get attention: "a key was
   pressed," "the timer ticked," "the mouse moved." We'll wire these up in the next
   two tasks.
3. **Software interrupts** — code can deliberately trigger one with the `int`
   instruction. We use this to *test* the system safely.

When any interrupt fires, the CPU looks up its number in the **IDT** to find which
function to jump to. The IDT is just an array of 256 entries: "interrupt #N → run the
code at this address." Building and registering that table is this task.

## How a single interrupt actually flows

This is the heart of it. When, say, interrupt #3 fires:

```
CPU pauses your code
  → looks up entry #3 in the IDT
  → jumps to the tiny assembly "stub" registered there (isr3 in isr_stubs.asm)
     → the stub saves all the CPU registers onto the stack
     → calls our ONE C handler, isr_handler(), passing it those saved registers
        → our C code decides what to do (print a message, fix things, or halt)
     → the stub restores the registers
  → `iret` instruction: CPU resumes exactly where it left off, as if nothing happened
```

The clever part is that **all 32 exception stubs are nearly identical** and funnel
into a **single** C function. We don't write 32 separate handlers in C; we write 32
trivial 3-line assembly stubs (generated by a macro) that all jump to one shared
routine. That shared routine saves state uniformly and hands a tidy struct of saved
registers to our C handler. After the handler returns, state is restored and `iret`
("interrupt return") resumes the interrupted program.

> **Why assembly for the stubs?** When an interrupt hits, the CPU is in a fragile
> in-between state — registers hold the interrupted program's values, and C can't run
> until those are safely saved. The stubs do that saving by hand, *then* it's safe to
> call C. This save/restore-around-C pattern is universal in OS interrupt handling.

### A subtle detail we had to get exactly right

Some exceptions make the CPU automatically push an extra "error code" onto the stack;
others don't. To keep our shared routine simple, the stubs for the ones that *don't*
push a dummy zero, so the stack always looks identical regardless of which exception
fired. The exact list of error-code-pushing exceptions (vectors 8, 10–14, 17, 21, 30)
has to be precisely correct — one wrong entry and the saved-register layout would be
off by 4 bytes, corrupting everything. (Our reviewer verified this list and the
register layout byte-for-byte.)

## Proving it works — safely

We can't exactly trigger a real divide-by-zero to test this without risk. So we use a
**deliberate, harmless software interrupt**: the `int $0x3` instruction (a
"breakpoint"). The kernel fires it on purpose right after setting up the IDT.

Our C handler has a special case: if it sees interrupt #3, it prints `ISR3_OK` and
**returns normally** (resumes the kernel). Seeing `ISR3_OK` on the log — *and* the
kernel continuing happily afterward — proves the entire round trip works: CPU paused,
found our handler via the IDT, ran our C code, restored state, and resumed. For any
*other* exception (a real bug), the handler instead prints the exception's name and
**halts** the CPU, which is exactly what you want — a crash that tells you what broke
instead of silently misbehaving.

## What about hardware interrupts (IRQs)?

This task also lays the groundwork for hardware interrupts: there are 16 more stubs
(for IRQs 0–15) and a second shared routine (`irq_handler`) plus a registry where
future drivers can plug in their handler ("call *this* function when the keyboard
fires"). But the hardware IRQs aren't actually switched on yet — that needs the next
piece, the PIC. So this task builds the catcher's mitt; the next two tasks throw the
balls.

> One loose end you might notice in the code: `irq_handler` calls `pic_send_eoi`,
> which doesn't exist yet (it belongs to the next task). To keep this step building on
> its own, we added a temporary empty placeholder for it, clearly marked, that the
> next task replaces with the real thing. This is a normal way to keep work
> independently testable when there's a forward dependency.

- ⏭️ Next: the **PIC** — remap and configure the chip that delivers hardware

**Jargon recap:** *interrupt* = "stop and handle this now." *exception* = an
interrupt the CPU raises on an error (vectors 0–31). *IRQ* = a hardware interrupt from
a device. *IDT* = the 256-entry table mapping interrupt number → handler address.
*ISR* (Interrupt Service Routine) = the handler that services an interrupt. *stub* =
the tiny assembly shim that saves state then calls our C handler. *iret* = the
instruction that resumes the interrupted code. *error code* = extra data some
exceptions push, which our stubs normalize away.
